# Business Case: HeavyBid Setup Project

## 1. Executive Summary

The HeavyBid Setup Project is intended to establish a structured, reliable, and scalable estimating environment inside HCSS HeavyBid. The goal is not simply to “get HeavyBid working,” but to configure it in a way that supports accurate estimating, repeatable workflows, future HeavyJob integration, and long-term operational visibility.

MICA is currently in a transition period where estimating knowledge, production assumptions, cost structures, quote handling, and project setup logic need to be captured in a more deliberate system. HeavyBid can become the central estimating platform for this work, but only if the initial setup is treated as a business process and data governance project, not just a software configuration task.

The business case for this project is based on four primary needs:

1. Improve consistency and confidence in estimates.
2. Reduce rework caused by unclear setup, disconnected codebooks, or manual workarounds.
3. Support the future HeavyBid-to-HeavyJob workflow.
4. Create a foundation for reporting, analytics, and continuous improvement.

---

## 2. Background

MICA is implementing HCSS HeavyBid as part of a broader effort to modernize estimating and project setup workflows. HeavyBid has the potential to improve bid preparation, quote management, crew and production modeling, and downstream integration with HeavyJob.

However, the value of HeavyBid depends heavily on the quality of the initial setup. Key system components such as bid item codebooks, activity codes, material codes, quote folders, resource structures, cost codes, and standard assemblies must be designed intentionally.

The project is currently at a critical stage: enough system work has been completed to demonstrate potential, but the setup still requires governance, documentation, validation, and alignment before it can be relied on as a scalable operating system for estimating.

---

## 3. Problem Statement

MICA needs a standardized HeavyBid setup that supports accurate estimating, consistent workflows, and future integration with HeavyJob. Without a structured setup process, the organization risks building estimates on inconsistent assumptions, unclear coding structures, duplicate data, and undocumented workarounds.

The immediate challenge is that HeavyBid can appear functional before it is actually stable. A bid can be built, priced, and exported while still relying on weak structure underneath. This creates a risk that early success may hide deeper setup issues that become more expensive to fix later.

Key problems include:

* Inconsistent or incomplete codebook structures.
* Lack of clear governance over bid items, activities, resources, materials, and quote folders.
* Risk of misalignment between estimating detail and accounting or job-costing structures.
* Limited documentation of setup decisions and business logic.
* Potential disconnect between HeavyBid setup and future HeavyJob use.
* Heavy reliance on individual knowledge instead of repeatable system logic.

---

## 4. Business Need

The business needs a HeavyBid environment that can support both immediate estimating work and long-term operational improvement.

The system should allow MICA to:

* Build estimates consistently across estimators and project types.
* Capture production assumptions, crews, resources, and quote logic in a reusable structure.
* Reduce manual rework during bid preparation.
* Improve confidence in bid review discussions.
* Support eventual transfer from HeavyBid into HeavyJob.
* Preserve institutional knowledge inside the system instead of relying only on individual memory.
* Create a data foundation for future analytics, reporting, and process improvement.

---

## 5. Project Objectives

The primary objective is to establish a governed HeavyBid setup that is usable, documented, maintainable, and aligned with MICA’s estimating and project delivery workflows.

Specific objectives include:

* Define the base document structure for the HeavyBid Setup project.
* Develop initial governance documents for codebooks, resources, quote setup, and estimating workflows.
* Review the current state of the HeavyBid environment.
* Identify setup gaps, risks, and misalignments.
* Develop an executive summary of the current project status.
* Establish a repeatable approach for future HeavyBid configuration decisions.
* Support parallel documentation for the broader HCSS Implementation project.
* Create a foundation for future HeavyBid-to-HeavyJob integration.

---

## 6. Scope

### In Scope

The project includes:

* HeavyBid setup documentation.
* Codebook structure review.
* Bid item, activity, material, and resource organization.
* Quote folder and quote system setup logic.
* Standard assembly and calculation approach.
* Governance documentation.
* Executive summary development.
* Current-state assessment.
* Risk identification.
* HeavyBid-to-HeavyJob readiness considerations.

### Out of Scope

The project does not currently include:

* Full production rollout of HeavyJob.
* Complete redesign of company accounting structures.
* Final approval of all estimating standards.
* Replacement of existing leadership decision-making processes.
* Full automation of all estimate types.
* Final enterprise reporting or analytics implementation.

Those areas may become future phases once the HeavyBid foundation is stable.

---

## 7. Recommended Approach

The recommended approach is to treat the HeavyBid Setup Project as a phased implementation with documentation and governance built alongside system configuration.

### Phase 1: Current-State Discovery

Review the existing HeavyBid environment to determine what has already been configured, what is usable, what needs revision, and what remains undefined.

Deliverables:

* Current-state summary.
* Known setup gaps.
* Risk list.
* Initial executive summary.

### Phase 2: Governance and Structure

Develop documentation that defines how HeavyBid setup decisions should be made and maintained.

Deliverables:

* Data governance document.
* Codebook governance notes.
* Resource and activity structure guidance.
* Quote folder setup guidance.
* Naming and numbering standards.

### Phase 3: Configuration Refinement

Use the governance structure to refine HeavyBid setup components.

Deliverables:

* Cleaned codebook structures.
* Improved activity and resource alignment.
* Material and quote folder organization.
* Standardized assembly patterns.
* Documented setup decisions.

### Phase 4: Validation

Test the setup against real estimating scenarios.

Deliverables:

* Pilot estimate review.
* HeavyBid workflow validation.
* Identified corrections.
* Readiness assessment for broader use.

### Phase 5: Integration Readiness

Evaluate whether the HeavyBid setup is suitable for future HeavyJob integration.

Deliverables:

* HeavyBid-to-HeavyJob readiness notes.
* Cost code alignment review.
* Transfer workflow considerations.
* Remaining integration risks.

## 8. Expected Benefits

### Operational Benefits

* More consistent estimating workflows.
* Reduced confusion during bid setup.
* Faster onboarding for estimators or support staff.
* Better reuse of assemblies, resources, and quote logic.
* Less dependence on undocumented tribal knowledge.

### Financial Benefits

* Reduced risk of missed costs.
* Improved quote comparison and vendor pricing control.
* Better visibility into labor, equipment, material, and subcontractor assumptions.
* Improved estimate review quality before bid submission.
* Potential reduction in rework after bid day.

### Strategic Benefits

* Stronger foundation for HeavyJob integration.
* Better long-term data quality.
* Improved ability to analyze historical estimating assumptions.
* Better alignment between estimating, operations, and accounting.
* Increased organizational confidence in HCSS as a core business platform.

---

## 9. Risks of Not Proceeding

If the HeavyBid setup is rushed or left undocumented, MICA may experience:

* Estimates that are technically functional but structurally inconsistent.
* Difficulty troubleshooting setup problems later.
* Increased resistance from users due to confusion or unreliable workflows.
* Poor data transfer into HeavyJob.
* Duplicate or conflicting codebook entries.
* Inconsistent production assumptions between estimates.
* Loss of setup knowledge when individual users are unavailable.
* More expensive cleanup after the system is already in active use.

The largest risk is premature go-live: HeavyBid may appear ready because estimates can be created, while foundational setup issues remain unresolved beneath the surface.

---

## 10. Constraints and Dependencies

### Constraints

* Business pressure to make the system usable quickly.
* Existing setup decisions may need to be reviewed without disrupting progress.
* Leadership may prefer immediate results over deeper analysis.
* Estimating workflows must continue while setup work is performed.
* HeavyBid structure must balance estimating detail with practical job-costing needs.

### Dependencies

* Access to current HeavyBid setup.
* Input from estimating leadership.
* Input from operations and project management.
* Understanding of accounting/job-costing expectations.
* HCSS support or documentation where needed.
* Availability of real estimates for testing.
* Clear decision authority for codebook and workflow standards.

---

## 11. Stakeholders

Primary stakeholders include:

* Estimators.
* Company leadership.
* Operations/project management.
* Accounting/job-costing personnel.
* HeavyJob users or future HeavyJob users.
* System administrators.
* Vendors and subcontractors indirectly through quote workflows.

Each stakeholder group has a different interest in the setup. Estimators need speed and accuracy. Operations needs usable project handoff. Accounting needs cost structure alignment. Leadership needs confidence that the system supports business goals.

---

## 12. Success Criteria

The project should be considered successful when:

* HeavyBid setup decisions are documented and understandable.
* Core codebooks follow a consistent structure.
* Bid items, activities, resources, materials, and quotes are logically organized.
* Standard assemblies can be reused and reviewed.
* Estimators can build estimates with fewer manual workarounds.
* Leadership can review estimates with greater confidence.
* Known setup risks are documented and actively managed.
* The system is positioned for future HeavyJob integration.
* The project has a clear executive summary and current-state assessment.

## 14. Recommendation

MICA should continue the HeavyBid Setup Project as a structured business improvement initiative. The project should not be treated as a simple software setup task. HeavyBid is becoming part of the company’s estimating infrastructure, and its configuration will affect bid quality, project handoff, cost tracking, and future analytics.

The recommended action is to formalize the project around documentation, governance, validation, and phased refinement. This will allow MICA to continue making progress while reducing the risk of building long-term workflows on unstable setup logic.

The immediate priority should be to establish the base documentation structure, complete a current-state assessment, and develop executive summaries for both the HeavyBid Setup project and the broader HCSS Implementation project. This will provide leadership with a clear view of where the projects stand, what value they are creating, and what decisions are needed next.

---

## 15. Draft Decision Statement

MICA should approve continued structured development of the HeavyBid Setup Project, including governance documentation, current-state assessment, codebook refinement, and HeavyBid-to-HeavyJob readiness planning. This work will reduce implementation risk, improve estimating consistency, and establish the foundation for long-term HCSS value across estimating, operations, and reporting.
