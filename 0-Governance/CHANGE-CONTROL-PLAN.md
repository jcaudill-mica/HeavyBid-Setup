---

## 1. Purpose

The purpose of this Change Control Plan is to define how changes to the HeavyBid Setup project will be identified, evaluated, approved, documented, and communicated.

Because the HB Setup project affects estimating workflows, codebooks, resource structures, activity structures, bid item mappings, reporting assumptions, and future HeavyBid-to-HeavyJob integration, uncontrolled changes could create confusion, rework, inconsistent estimating practices, or implementation risk.

This plan establishes a consistent process for managing changes while allowing the project team to continue learning, refining, and improving the setup as business needs become clearer.

---

## 2. Scope

This Change Control Plan applies to changes affecting the configuration, documentation, assumptions, governance, or implementation direction of the HB Setup project.

Change control applies to, but is not limited to:

* HeavyBid codebook structure
* Bid Item Codebook design
* Activity Codebook design
* Material, labor, equipment, subcontractor, and crew resource structures
* Estimate templates and library estimates
* Assemblies and reusable estimate components
* Quote folder structures and vendor quote workflows
* Cost code structures and Alt Cost Code mappings
* HeavyBid-to-HeavyJob transfer assumptions
* Reporting and analytics assumptions
* Project governance documents
* Implementation milestones or priorities
* Roles, responsibilities, or approval expectations

This plan does not apply to minor working notes, informal research, sandbox experimentation, or temporary drafts unless they are promoted into the official project documentation or setup baseline.

---

## 3. Change Control Objectives

The objectives of the change control process are to:

* Preserve a clear record of key project decisions and changes
* Prevent uncontrolled changes from creating downstream setup issues
* Ensure changes are evaluated for estimating, operations, accounting, and reporting impact
* Support alignment between HeavyBid setup and future HeavyJob integration
* Provide visibility to leadership and affected users
* Maintain version control over project documents and configuration assumptions
* Distinguish between exploration, recommendation, approval, and implementation
* Reduce the risk of premature go-live decisions based on incomplete setup

---

## 4. Change Categories

Changes should be classified by type to determine the appropriate level of review.

### 4.1 Administrative Changes

Administrative changes are low-risk updates that do not affect project direction or system configuration.

Examples include:

* Formatting corrections
* Typographical corrections
* Minor wording improvements
* Updated document dates
* Clarifying language that does not change intent

**Approval Level:** Project Owner or Document Owner
**Documentation Required:** Document change log entry, if part of a controlled document

---

### 4.2 Documentation Changes

Documentation changes affect official project documents but may or may not affect implementation.

Examples include:

* Updates to the Project Charter
* Updates to the Communication Plan
* Updates to the Risk Register
* Updates to the Decision Log
* Updates to process documentation
* Addition of new governance documents

**Approval Level:** Project Owner, with stakeholder review as needed
**Documentation Required:** Version update and document change log entry

---

### 4.3 Configuration Changes

Configuration changes affect the structure, setup, or behavior of HeavyBid.

Examples include:

* Revising Activity Codebook structure
* Changing Material Codebook organization
* Modifying crew or equipment resource structures
* Changing bid item mapping logic
* Revising quote folder organization
* Updating estimate templates
* Changing assemblies or production logic
* Modifying cost code or Alt Cost Code structure

**Approval Level:** Project Owner and affected functional stakeholder
**Documentation Required:** Change Request entry, Decision Log entry if approved, and updated setup documentation

---

### 4.4 Process Changes

Process changes affect how estimators, schedulers, managers, or other users will work within the HeavyBid ecosystem.

Examples include:

* New estimating workflow
* New handoff process from estimating to scheduling
* New quote management process
* New review or approval process
* New project setup process
* Changes to estimate closeout procedures

**Approval Level:** Project Sponsor or designated leadership representative
**Documentation Required:** Change Request entry, communication to affected users, updated process documentation

---

### 4.5 Strategic or Integration Changes

Strategic or integration changes affect project direction, implementation strategy, system integration, or organizational alignment.

Examples include:

* Changes to go-live timing
* Changes to HeavyBid-to-HeavyJob integration assumptions
* Changes to accounting or cost-code alignment strategy
* Changes to executive priorities
* Changes to implementation scope
* Changes that affect multiple departments

**Approval Level:** Project Sponsor and key stakeholders
**Documentation Required:** Formal Change Request, Decision Log entry, Risk Register review, and stakeholder communication

## 5. Change Request Process

The change control process follows six basic steps.

### Step 1: Identify the Change

A potential change may be identified during setup work, document review, stakeholder discussion, testing, estimating exercises, or leadership review.

The person identifying the change should capture:

* What is being changed
* Why the change is needed
* What problem or opportunity the change addresses
* What area of the project may be affected

---

### Step 2: Document the Change Request

A change request should be entered into the Change Request Log.

At minimum, each request should include:

* Change Request ID
* Date submitted
* Submitted by
* Change title
* Change description
* Reason for change
* Change category
* Estimated impact
* Recommended action
* Current status

---

### Step 3: Assess Impact

The project team should assess the proposed change before approval.

Impact assessment should consider:

* Estimating impact
* HeavyBid setup impact
* HeavyJob integration impact
* Accounting or cost-code impact
* Reporting and analytics impact
* Training impact
* Documentation impact
* Schedule impact
* Risk impact
* Rework required
* Stakeholders affected

For larger changes, the Risk Register should be reviewed to determine whether the change introduces a new risk or modifies an existing risk.

---

### Step 4: Review and Approve

The appropriate approval authority reviews the change based on the change category.

Possible decisions include:

* Approved
* Approved with modifications
* Deferred
* Rejected
* Needs more analysis

Approved changes should be added to the Decision Log when they affect project direction, configuration, process, or implementation assumptions.

---

### Step 5: Implement the Change

Once approved, the change may be implemented in the appropriate project area.

Implementation may include:

* Updating HeavyBid configuration
* Updating project documents
* Revising process maps
* Updating codebook documentation
* Adjusting templates or assemblies
* Communicating the change to users
* Testing the change in a sample estimate
* Updating related logs or registers

---

### Step 6: Verify and Close

After implementation, the change should be verified.

Verification may include:

* Confirming the configuration was updated correctly
* Confirming documentation reflects the approved change
* Testing the change in HeavyBid
* Confirming affected stakeholders were notified
* Confirming related risks or decisions were updated

Once verified, the change request may be marked as closed.

## 6. Change Request Status Definitions

| Status                      | Meaning                                               |
| --------------------------- | ----------------------------------------------------- |
| Draft                       | Change has been identified but not fully documented   |
| Submitted                   | Change request has been entered for review            |
| Under Review                | Change is being evaluated for impact                  |
| Approved                    | Change has been approved for implementation           |
| Approved with Modifications | Change is approved but with revisions or limitations  |
| Deferred                    | Change is valid but postponed                         |
| Rejected                    | Change will not be implemented                        |
| Implemented                 | Change has been completed but not yet verified        |
| Closed                      | Change has been implemented, verified, and documented |

## 8. Approval Guidelines

Not every change requires the same level of approval. The following guidelines should be used to determine the appropriate review path.

| Change Type                                | Approval Needed                                              |
| ------------------------------------------ | ------------------------------------------------------------ |
| Typo, formatting, minor clarification      | Document Owner                                               |
| Document structure or content update       | Project Owner                                                |
| HeavyBid configuration change              | Project Owner and affected functional stakeholder            |
| Workflow or process change                 | Project Owner and Project Sponsor or department lead         |
| Cost code, accounting, or reporting change | Project Owner, accounting stakeholder, and sponsor as needed |
| HeavyBid-to-HeavyJob integration change    | Project Sponsor and affected department stakeholders         |
| Go-live, scope, or milestone change        | Project Sponsor                                              |

## 12. Version Control Expectations

Controlled documents should include a version number and change log.

Recommended versioning approach:

| Version Type | Use Case                                                |
| ------------ | ------------------------------------------------------- |
| v0.x         | Draft versions before formal approval                   |
| v1.0         | First approved baseline version                         |
| v1.x         | Minor approved updates                                  |
| v2.0         | Major restructuring or significant scope/process change |

Document versions should be updated when changes affect meaning, structure, approval status, or project direction.

Minor typo corrections do not require a version change unless the document is already under formal control.

## 13. Communication of Approved Changes

Approved changes should be communicated based on impact.

Communication methods may include:

* Project meeting discussion
* Teams message or post
* Email summary
* Updated project document
* Updated decision log
* Updated process map
* OneNote project journal entry
* Direct communication to affected stakeholders

For significant changes, the communication should include:

* What changed
* Why it changed
* Who is affected
* When the change takes effect
* What action is required
* Where the updated documentation is located

## 14. Change Control Principles

The following principles should guide change control for the HB Setup project:

1. **Change is expected, but it must be visible.**
   The project is still developing, and learning will occur throughout the setup process.

2. **Configuration changes should not be made casually.**
   HeavyBid setup decisions can affect estimating consistency, reporting, and downstream integration.

3. **Decisions should be documented close to the time they are made.**
   Delayed documentation increases the chance of losing context.

4. **The reason for a change matters as much as the change itself.**
   Future users need to understand why decisions were made.

5. **Approved changes should connect back to project objectives.**
   Changes should improve alignment, usability, accuracy, reporting, or implementation readiness.

6. **Not all ideas are approved changes.**
   Exploration, testing, and discussion are allowed, but official setup changes should pass through the change control process.

## 15. Initial Known Change Control Concerns

The following areas should receive particular attention during the HB Setup project:

* Potential pressure to go live before setup is fully analyzed
* Changes to codebook structure after estimates have already been built
* Differences between estimating detail and accounting cost-code requirements
* HeavyBid-to-HeavyJob integration assumptions
* Resource naming and classification consistency
* Quote folder structure and vendor quote management
* Use of library estimates and reusable assemblies
* Differences between leadership expectations and practical estimating workflow
* Documentation drift as the setup evolves
* Informal decisions being made without being captured in the Decision Log
